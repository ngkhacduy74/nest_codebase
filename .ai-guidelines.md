# AI Development Guidelines

## Mục tiêu
File này định nghĩa các quy tắc và nguyên tắc mà AI phải tuân thủ khi làm việc với codebase để đảm bảo tính ổn định, nhất quán và tránh phá vỡ cấu trúc hiện có.

## Nguyên tắc cốt lõi

### 1. KHÔNG thay đổi cấu trúc code không cần thiết
- Chỉ sửa code khi có yêu cầu rõ ràng hoặc fix bug cụ thể
- KHÔNG tự động refactor, tái cấu trúc, hay "cải tiến" code không được yêu cầu
- Giữ nguyên pattern, architecture, và coding style hiện có
- Nếu thấy code "xấu" nhưng vẫn hoạt động, hãy để nguyên

### 2. Tuân thủ quy tắc TypeScript và NestJS
- Luôn maintain type safety
- Sử dụng dependency injection pattern của NestJS
- Giữ nguyên cấu trúc module/service/repository hiện có
- KHÔNG thêm dependencies mới không cần thiết

### 3. Quy tắc khi làm việc với Database
- KHÔNG tự động thay đổi Prisma schema
- KHÔNG tạo migration không cần thiết
- Giữ nguyên naming conventions của database
- Sử dụng repository pattern hiện có

### 4. Testing guidelines
- Chỉ sửa test khi có yêu cầu hoặc khi test bị break do thay đổi code
- Giữ nguyên structure và naming của test files
- KHÔNG tự động thêm test cases mới
- Maintain mock objects structure hiện có

### 5. Quy tắc về file và folder structure
- KHÔNG tạo file mới không được yêu cầu
- KHÔNG xóa file không được yêu cầu
- Giữ nguyên import/export structure
- Sử dụng absolute paths theo convention hiện có

## Quy trình làm việc

### Khi fix bug
1. Đọc kỹ error message và context
2. Tìm root cause trước khi sửa
3. Chỉ thay đổi code tối thiểu cần thiết
4. Verify fix không ảnh hưởng phần khác
5. KHÔNG refactor code xung quanh

### Khi thêm feature
1. Hỏi clarifying questions nếu không rõ yêu cầu
2. Làm theo pattern hiện có trong codebase
3. Sử dụng components/services đã có nếu possible
4. Giữ nguyên naming conventions
5. Test chỉ những phần mình thay đổi

### Khi làm việc với existing code
1. Đọc kỹ file trước khi edit
2. Hiểu purpose và context của code
3. Giữ nguyên coding style (indentation, naming, etc.)
4. KHÔNG "optimize" hoặc "modernize" code không cần thiết

## Red flags - Những điều KHÔNG BAO GIỜ làm

❌ **TUYỆT ĐỐI KHÔNG:**
- Tự động upgrade dependencies
- Thay đổi import statements không cần thiết
- Refactor function names hoặc variable names
- Thay đổi error handling structure
- Modify configuration files không được yêu cầu
- Thay đổi git hooks hoặc CI/CD setup
- Modify Docker configuration không cần thiết

✅ **LUÔN LUÔN:**
- Hỏi trước khi không chắc
- Đọc documentation có sẵn
- Làm theo existing patterns
- Test changes trước khi commit
- Giữ nguyên backward compatibility

## Emergency rules

Nếu AI không chắc về điều gì:
1. Dừng lại và hỏi user
2. Không tự quyết định
3. Ưu tiên safety over "improvement"
4. Khi doubt, don't change

## Examples

### Good practice:
```typescript
// Giữ nguyên structure hiện có
@Injectable()
export class ExistingService {
  constructor(
    @Inject(USER_REPOSITORY) private userRepo: UserRepository,
  ) {}
  
  // Chỉ thêm method mới khi được yêu cầu
  async newMethod(): Promise<void> {
    // Implementation theo pattern hiện có
  }
}
```

### Bad practice:
```typescript
// KHÔNG refactor không cần thiết
@Injectable() 
export class RenamedService { // ❌ Không đổi tên
  constructor(
    @Inject('USER_REPOSITORY') private repo: any, // ❌ Không đổi type
  ) {}
  
  // ❌ Không modernize code không cần thiết
  async existingMethod = async () => {
    // ...
  }
}
```

## Final reminder
**Goal là maintain stability, không phải show off coding skills.** Code hiện có đang hoạt động thì đừng động vào.
